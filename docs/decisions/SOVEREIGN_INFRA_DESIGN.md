# 소버린 인프라 사업 설계서 (재설계 v1.0)

> 작성 기준: 첨부 설계서 + 이 대화 맥락 + 코드 검증 결과만 사용. 공유지식 미사용.
> 설계 순서: **최종 목표 아키텍처를 먼저 확정** → 그 위에서 초기 사업이 어느 레이어를 임시 대체하는지 구별.

---

## 0. 설계 원칙 — 왜 이 순서인가

이전 설계는 "초기 사업을 시작하고, 수익이 생기면 자체화한다"는 순서였습니다. 이 순서는 틀렸습니다. 수익이 없으면 시작 자체가 불가능하기 때문입니다.

올바른 순서는 반대입니다.

> **최종 목표 아키텍처를 먼저 100% 확정한다. 그 아키텍처의 모든 레이어는 처음부터 "우리 것"으로 설계되어 있다. 단지 초기에는 일부 레이어의 내부 구현을 Vultr 관리형/외부 API로 임시 대체하여 채워 넣고, 시간이 지나면서 그 내부를 자체 엔진으로 갈아 끼운다.**

핵심 차이는 이것입니다. 고객이 보는 API, 우리가 소유하는 제어 평면, 데이터가 쌓이는 원장 — 이것들은 **첫날부터 최종 형태**입니다. 바뀌는 것은 오직 각 레이어 뒤에 숨은 실제 실행 엔진뿐입니다. 이 구조가 성립하려면 **포트-어댑터 경계**가 최종 아키텍처에 처음부터 박혀 있어야 합니다.

이것이 "초기 사업"과 "최종 목표"를 구별하는 유일하게 올바른 방법입니다. 두 개의 다른 사업이 아니라, **하나의 최종 사업이고 내부 엔진만 교체**되는 것입니다.

---

## 1. 최종 목표 아키텍처 — 전체 조감도

우리가 최종적으로 소유하는 것은 4개의 서비스 평면과 그것들을 묶는 1개의 제어 평면입니다.

```
                    [ 고객 / 고객의 최종 유저 ]
                              │
                              │  단일 API 엔드포인트 (첫날부터 우리 소유)
                              ▼
        ┌─────────────────────────────────────────────┐
        │  제어 평면 (Control Plane) — sovereign_core   │  ← 절대 외주/대체 안 함
        │  인증 · 정산원장 · 멱등성 · 라우팅 · 격리계약   │
        └───────┬─────────┬─────────┬─────────┬─────────┘
                │         │         │         │
         (포트) │  (포트) │  (포트) │  (포트) │  ← 어댑터 교체 지점
                ▼         ▼         ▼         ▼
        ┌───────────┐ ┌────────┐ ┌────────┐ ┌──────────┐
        │ L2 스토리지│ │L3 연산 │ │L4 AI   │ │ 중계/과금 │
        │ (Neon)    │ │(Modal) │ │(Bedrock│ │(OpenRouter)│
        │           │ │        │ │  )     │ │           │
        └───────────┘ └────────┘ └────────┘ └──────────┘
         각 레이어 = [ 포트(계약) ] + [ 어댑터(초기=관리형 / 최종=자체엔진) ]
```

**변하지 않는 것 (첫날 = 최종날):**
- 고객이 호출하는 API 계약
- 제어 평면(인증, 정산, 멱등성, 라우팅, 격리)
- 크레딧 원장과 마진 계산
- 포트 인터페이스 정의

**변하는 것 (초기 → 최종):**
- 각 포트 뒤의 어댑터 = 실제 실행 엔진

## 2. 4대 서비스 최종 자체 구현 설계

### L2 — 스토리지 평면 (Neon 대체)

**최종 목표가 제공하는 것:** 고객이 "브랜치"를 만들면 즉시 격리된 Postgres 호환 DB가 생기고, 스냅샷/타임트래블/CoW 복제가 되는 서버리스 DB.

| 구성요소 | 최종 자체 구현 | 역할 |
|---|---|---|
| Pageserver | 자체 구현 (Rust/Go) | 페이지 단위 스토리지, LSM 인덱스, CoW 브랜치 |
| Safekeeper | 자체 구현 | WAL 다중화, 내구성 보장 |
| Compute 노드 | Postgres 프로토콜 호환 실행기 | 고객 쿼리 실행 |
| 오브젝트 백엔드 | S3 호환 (자체 MinIO 클러스터 or Vultr Object) | 콜드 데이터 |

**포트 계약:** `StoragePort` — `CreateBranch`, `Snapshot`, `Restore(LSN)`, `AttachBranch`. 이 계약은 초기든 최종이든 동일합니다.

### L3 — 격리 연산 평면 (Modal 대체)

**최종 목표가 제공하는 것:** 고객 코드/함수를 밀리초 단위로 부팅하는 격리 샌드박스. 사용한 시간만 과금.

| 구성요소 | 최종 자체 구현 | 역할 |
|---|---|---|
| MicroVM 런타임 | Firecracker 자체 운영 | 강한 격리 + 빠른 부팅 |
| 웜풀 (Warm Pool) | 자체 구현 (warmpool.go 설계됨) | 콜드스타트 제거, 미리 부팅된 VM 유지 |
| 스냅샷 복원 | Firecracker snapshot | 상태 복원으로 부팅 시간 최소화 |
| 스케줄러 | 자체 제어 평면이 관장 | tenant/model 기반 노드 배치 |

**포트 계약:** `ComputePort` — `BootInstance`, `AttachBranch`, `Snapshot`, `Terminate`.

### L4 — AI 서빙 평면 (Bedrock 대체)

**최종 목표가 제공하는 것:** 고객이 등록한 OSS 모델 + LoRA를 고성능으로 서빙. 토큰 단위 과금.

| 구성요소 | 최종 자체 구현 | 역할 |
|---|---|---|
| 추론 엔진 | vLLM 자체 운영 | Continuous Batching |
| KV 캐시 관리 | PagedAttention | VRAM 단편화 제거 |
| 다중 LoRA | S-LoRA 주소 매핑 | 베이스 모델 1개 + LoRA N개 동시 서빙 |
| GPU 클러스터 | Vultr GPU 베어메탈 | 물리 연산 |

**포트 계약:** `InferencePort` — `Serve(modelID, loraID, prompt)`, `CountTokens`, `Cancel`. (신규 정의 필요)

### 중계/과금 평면 (OpenRouter 대체)

**최종 목표가 제공하는 것:** 외부 LLM + 자체 모델 + 고객 OSS 모델을 단일 엔드포인트에서 중계. 토큰 카운팅 + 마진 + 선차감.

| 구성요소 | 최종 자체 구현 | 역할 |
|---|---|---|
| 토큰 카운터 | Envoy 인라인 필터 자체 운영 | 스트리밍 중 실시간 토큰 계측 |
| 크레딧 원장 | 자체 구현 (ledger.go 완료) | 잔액, 선차감, 음수 방지 |
| 마진 엔진 | 자체 구현 (adapter_billing.go 완료) | 원가 + 마진 계산 |
| 모델 마켓 | 자체 구현 | 고객 OSS 모델 등록/벤치마크 |

**포트 계약:** `UpstreamLLMPort` (신규) + `PaymentPort` (완료).

## 3. 제어 평면 — 유일하게 절대 대체하지 않는 것

제어 평면은 초기든 최종이든 **100% 우리 것**입니다. 이것이 사업의 IP이자 해자(moat)입니다.

| 기능 | 현재 상태 (코드 검증됨) |
|---|---|
| 멱등성(이중과금 방지) | 원자적 예약 패턴 구현·검증 완료 (60/60 op=1) |
| 크레딧 원장(음수 방지) | 정수 회계 + 원자적 검사-차감 검증 완료 |
| 마진 계산(역마진 방지) | 검증 완료 (역마진 0건) |
| 원자적 온보딩(좀비 방지) | 커밋 순서 재배치로 수정·검증 완료 |
| 교차 테넌트 격리(RLS) | 검증 완료 (누출 0건) |
| 인증(토큰 위조) | Mock 한계 식별 — 실 IdP 교체 필요 |

**핵심:** 제어 평면은 이미 상당 부분 구현·검증되어 있습니다. 초기 사업이든 최종 목표든 이 코드는 그대로 씁니다.

## 4. 초기 사업 — 최종 아키텍처의 어느 레이어를 임시 대체하는가

이제 핵심입니다. 위의 최종 아키텍처는 그대로 두고, **각 레이어 뒤의 어댑터만 초기용으로 채워 넣습니다.** 고객이 보는 것, 우리가 소유하는 제어 평면은 첫날부터 최종 형태입니다.

### 대체 원칙 — 무엇을 자체 소유하고 무엇을 임시 대체하는가

레이어를 세 종류로 분류합니다.

**(1) 첫날부터 자체 소유 — 절대 대체 안 함 (= 우리의 IP)**
- 제어 평면 전체: 인증, 정산 원장, 멱등성, 라우팅, 격리 계약, 마진 엔진
- 이유: 이것이 사업의 해자다. 여기에 고객 데이터와 과금 정보가 쌓이고, 여기가 Lock-in을 만든다. 외주하면 사업이 아니라 대리점이 된다.

**(2) 초기엔 관리형/외부로 대체 — 나중에 자체 엔진으로 교체 (= 어댑터)**
- L2 스토리지 실행 엔진, L3 연산 실행 엔진, L4 AI 실행 엔진
- 이유: 이것들은 자체 구축에 시간과 하드웨어가 필요하다. 하지만 포트 계약 뒤에 숨기면 고객은 차이를 모른다.

**(3) 초기든 최종이든 계속 외부에 맡김 — 자체화 안 함 (= 상용 서비스 활용)**
- 결제 PG(카드 결제), DNS, 인증서 발급, 기본 이메일
- 이유: 이건 우리 IP가 아니고, 자체화해도 경쟁력이 안 된다.

### 레이어별 초기 대체 매핑

| 레이어 | 포트 (첫날 확정) | 초기 어댑터 (지금) | 최종 어댑터 (자체) | 고객이 느끼는 차이 |
|---|---|---|---|---|
| 제어 평면 | — | 자체 (sovereign_core) | 자체 (동일) | 없음 (항상 우리 것) |
| L2 스토리지 | `StoragePort` | Vultr 관리형 Postgres | 자체 Pageserver | 없음 (API 동일) |
| L3 연산 | `ComputePort` | Vultr VM + 컨테이너 | 자체 Firecracker 웜풀 | 부팅 속도만 개선됨 |
| L4 AI 서빙 | `InferencePort` | 외부 LLM API 직결 | 자체 vLLM 클러스터 | 마진율만 개선됨 |
| 중계/과금 | `UpstreamLLMPort` + `PaymentPort` | 외부 API 중계 + PG | 자체 모델 + 볼륨딜 | 없음 |

**결정적 포인트:** L2/L3/L4의 초기 어댑터는 "임시방편"이 아니라 **최종 아키텍처의 정식 첫 번째 어댑터**입니다. 포트 계약이 동일하므로, 자체 엔진이 준비되면 어댑터만 바꿔 끼우고 제어 평면 코드는 한 줄도 안 바뀝니다. 이것은 이미 코드로 증명됐습니다 — MockStorage → FileStorage 교체 시 제어 평면 무변경으로 동작(T6 검증 완료).

## 5. 초기 사업이 "처음부터" 제공하는 것

초기라고 해서 반쪽짜리가 아닙니다. 최종 아키텍처의 계약을 그대로 제공합니다.

**초기 사업 출시 시점에 고객이 받는 것:**
1. 단일 API 엔드포인트로 외부 LLM(GPT/Claude/DeepSeek 등) + 고객 등록 OSS 모델 중계
2. 브랜치형 DB (초기엔 관리형 Postgres 위에 브랜치 추상화)
3. 격리 연산 샌드박스 (초기엔 컨테이너)
4. 크레딧 지갑 + 선차감 + 정확한 토큰 과금 (자체 원장, 첫날부터 최종 형태)
5. 마진이 붙은 투명한 청구서 (자체 마진 엔진)

**초기 사업의 수익원:**
- 외부 API 중계 마진 (원가 + 5~10%)
- 관리형 인프라 재판매 마진
- 이 매출로 자체 엔진 구축 자본을 축적 (수익이 시작 조건이 아니라, 자체화 가속 연료)

**중요:** 초기 사업 시작에 필요한 것은 "수익"이 아니라 "최소 자본으로 확보하는 초기 인프라 + 제어 평면 코드"입니다. 제어 평면은 이미 있습니다. 초기 인프라는 Vultr 관리형으로 월 $200~500 수준에서 시작 가능합니다.

## 6. 두 트랙을 잇는 다리 — 포트 계약이 전부다

```
   [ 초기 사업 ]                        [ 최종 목표 ]
        │                                    │
   같은 API 계약 ──────────────────────── 같은 API 계약
        │                                    │
   같은 제어 평면 ────────────────────── 같은 제어 평면
        │                                    │
   같은 포트 인터페이스 ──────────────── 같은 포트 인터페이스
        │                                    │
   초기 어댑터                           자체 엔진 어댑터
   (Vultr 관리형/외부 API)      ───교체──▶  (Pageserver/Firecracker/vLLM)
```

**이 다리가 성립하려면 지금 당장 해야 할 유일한 아키텍처 작업:** 4개 포트(`StoragePort`, `ComputePort`, `InferencePort`, `UpstreamLLMPort`)를 최종 아키텍처 기준으로 **완전하게 정의**하는 것. 이 중 `StoragePort`, `PaymentPort`는 이미 있고, `InferencePort`, `UpstreamLLMPort`는 신규 정의가 필요합니다.

## 7. 당장 할 일 — 검증 포함

당장 할 일은 "최종 아키텍처의 뼈대(포트 계약 + 제어 평면)를 완성하는 것"입니다. 이것은 하드웨어 없이 지금 전부 가능하며, 여기서 만든 것은 초기든 최종이든 그대로 쓰입니다.

### 7.1 이미 완료·검증된 것 (재사용 자산)

| 항목 | 상태 | 검증 결과 |
|---|---|---|
| 제어 평면 스켈레톤 | 완료 | 빌드/vet/관통 통과 |
| 멱등성(이중과금 방지) | 완료 | 60/60 op=1, 실서버 50발→1 |
| 크레딧 원장(음수 방지) | 완료 | 동시 20건→10 성공, 잔액 0 |
| 마진 엔진(역마진 방지) | 완료 | 전 조합 sell>cost |
| 원자적 온보딩 | 완료 | 좀비 org 0 |
| RLS 격리 | 완료 | 교차 누출 0 |
| StoragePort 실물 어댑터 | 완료 | 디스크 영속·동시 100건 무손실 |

이 자산들은 초기 사업과 최종 목표 양쪽 모두의 공통 뼈대입니다.

### 7.2 지금 당장 해야 할 일 (하드웨어 불필요)

| 우선 | 할 일 | 이유 | 검증 방법 |
|---|---|---|---|
| 1 | **`InferencePort` 신규 정의** | L4가 포트 없이는 어댑터 교체 불가. 최종 아키텍처의 마지막 빈 계약 | 인터페이스 정의 + mock 어댑터로 관통 테스트 |
| 2 | **`UpstreamLLMPort` 신규 정의** | 중계 사업의 핵심 계약. 외부 API↔자체 모델 전환 지점 | 정의 + mock 관통 테스트 |
| 3 | **토큰 카운팅 정합성 검증** | 중계 과금의 정확성. 스트리밍 중 토큰 계측이 원장과 일치하는가 | 동시 스트리밍 시 토큰 합계 = 과금액 테스트 |
| 4 | **인증 실 IdP 어댑터** | 현재 Mock은 토큰 위조 가능(T9-A 결함). 초기 사업 출시 전 필수 | 서명 검증 JWT로 위조 토큰 차단 테스트 |
| 5 | **선차감 → 실패 환불 경로** | 원장이 오퍼레이션 경로에 아직 미연결. 잔액 부족 시 402, 실패 시 환불 | 잔액 부족 거부 + 실패 롤백 테스트 |

## 8. 핵심 결론

**설계 순서를 바로잡았습니다.** 최종 목표 아키텍처가 먼저이고, 초기 사업은 그 아키텍처의 어댑터를 임시 구현으로 채운 상태입니다. 두 개의 다른 사업이 아니라 **하나의 사업이고 내부 엔진만 교체**됩니다.

**변하지 않는 것 (첫날 = 최종날):** 고객 API, 제어 평면, 정산 원장, 마진 엔진, 포트 계약. 이것이 IP이고 해자입니다. 이미 대부분 구현·검증되어 있습니다.

**변하는 것 (초기 → 최종):** L2/L3/L4 어댑터 뒤의 실행 엔진. 관리형/외부 API → 자체 Pageserver/Firecracker/vLLM. 포트 계약이 동일하므로 제어 평면 무변경 교체가 가능하며, 이것은 코드로 증명됐습니다(T6).

**초기 사업 시작 조건은 "수익"이 아닙니다.** 제어 평면(이미 있음) + 최소 인프라(월 $200~500) + 4개 포트 완성(지금 당장 가능)입니다. 수익은 시작 조건이 아니라 자체화를 가속하는 연료입니다.

**당장 할 일:** 최종 아키텍처의 마지막 빈 계약인 `InferencePort`, `UpstreamLLMPort`를 정의하고, 토큰 카운팅·인증·선차감 환불 경로를 검증하는 것. 전부 하드웨어 없이 지금 가능하며, 여기서 만든 것은 최종까지 그대로 쓰입니다.
