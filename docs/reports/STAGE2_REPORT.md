# Stage 2 검증 결과 보고서 — 정산 정합성 (Money Correctness)

> 근거: 첨부 코드 + 이 세션의 실제 실행 결과만 사용. 외부 공유지식 미사용.
> 실행 환경: Go 1.22.5 / linux-amd64 / `go test`(±`-race`)
> 대상: `sovereign_core`(control-plane mock 스켈레톤, v0.2) + 신규 `billing_correctness_test.go`
> 원칙: 기존 코드는 **한 줄도 수정하지 않음.** 테스트는 관측 도구이며, 결함이 있어도 코드를 고치지 않고 "숫자로 드러내는" 것이 목적.

---

## 1. 한눈에 보는 결과

| 테스트 | 명제 | 결과 | 실측 수치 |
|---|---|---|---|
| **T1** 멱등성 동시성 | 같은 Idempotency-Key로 동시 200발 → 실제 생성 op는 1개여야 | **실패(결함 재현)** | 50회 중 2회 이중생성, op=3·op=5 관측, **결함률 4%** |
| **T2** 교차 테넌트 격리 | org A 토큰으로 org B 자원 조회 → 404 | **통과** | 20/20 통과, HTTP 404 + store RLS 필터 정상 |
| **T3** 미터링 원장 정합성 | 10 org × 20 프로젝트 동시 → org별 정확히 20건 | **통과** | 20/20 통과, 원장 합계 200/200 정확 |
| **T4** 원자적 온보딩(장애주입) | KMS/billing 실패 시 org 부분생성 0 | **통과** | KMS·billing 실패 모두 org 0건 잔존 |

**결론:** 4개 정합성 명제 중 **3개는 현재 설계에서 이미 견고**하고, **1개(T1 멱등성)는 재현 가능한 이중과금 결함**이 실재합니다. 이것이 설계서의 "정합성 물리적 0%"라는 서사가 코드 레벨에서 실제로는 성립하지 않는다는 **물리적 증거**입니다.

---

## 2. T1 — 이중과금 결함: 재현 가능한 물리적 증거

### 2.1 실측 데이터

동일 `Idempotency-Key` 하나로 200개 요청을 최대한 동시에 발사하는 실험을 반복:

| 실행 조건 | 반복 | 이중생성 발생 | 관측된 최대 op 수 |
|---|---|---|---|
| 캐시 무효화(`-count=1`) | 50회 | **2회 (4%)** | op=5 (하나의 멱등키로 5개 operation) |
| `-race` 활성 | 다회 | 재현됨 | op=3 |

정상이라면 같은 멱등키에 대해 실제 생성 operation은 **항상 정확히 1개**여야 합니다. 그러나 실측에서는 하나의 멱등키로 **3개, 5개의 독립 operation**이 생성되었습니다. 각 operation은 각자 `CreateBranch`를 호출하고 각자 `branch_ops` 미터링 이벤트를 적재하므로, **1회 과금 계약이 3~5회 과금으로 부풀려지는** 것과 동일합니다.

### 2.2 근원 (코드 특정)

`handlers.go`의 멱등성 래퍼 `startOp`:

```
key := r.Header.Get("Idempotency-Key")
if prev, ok := s.app.store.idempLookup(key); ok {   // ① 조회
    ... return 202 replay
}
op := make(...)                                      // ② op 생성(부수효과: 리소스/미터링)
s.app.store.idempStore(key, op.ID)                   // ③ 기록
```

`①조회 → ②생성 → ③기록`이 **세 개의 분리된 store 호출**로 이루어져 있고, 그 사이에 잠금이 없습니다. 이는 전형적인 **TOCTOU(Time-Of-Check to Time-Of-Use)** 결함입니다. 동시 요청 A·B·C가 모두 ①에서 "없음"을 확인한 뒤 각자 ②를 실행하면, 멱등키가 하나여도 operation이 여러 개 생성됩니다. `Store` 내부의 각 메서드(`idempLookup`, `put`, `idempStore`)는 개별적으로는 뮤텍스로 보호되지만, **"조회하고 없으면 생성한다"는 복합 연산 전체가 원자적이지 않기 때문**에 락이 있어도 소용이 없습니다.

### 2.3 왜 100%가 아니라 4%인가

이중생성은 여러 요청이 ②를 실행하기 전에 ①을 통과해야 발생하므로, 스케줄러가 그 좁은 창(window)에 여러 고루틴을 몰아넣을 때만 나타납니다. 그래서 **대부분(96%)은 정상처럼 보이고 가끔(4%) 터지는** 비결정적 결함입니다. 이런 유형이 가장 위험합니다 — 개발/스테이징에서는 "잘 되는데?"로 보이다가 **프로덕션의 고부하 순간에만 재무 손실을 일으키기** 때문입니다. 설계서가 목표한 30,000~50,000 RPS에서는 이 창이 훨씬 자주 열립니다.

### 2.4 의미(해석)

- 이 결함은 **코드가 틀려서**가 아니라, **"정합성은 당연히 된다"고 가정하고 넘어갔기** 때문에 존재합니다. Stage 2의 존재 이유가 바로 이것입니다.
- 수정 자체는 어렵지 않습니다(멱등 예약을 단일 원자 연산으로: 예 — 잠금 하 `LoadOrStore`, 또는 DB `INSERT ... ON CONFLICT`로 멱등키에 유니크 제약). **다만 이번 작업의 규칙상 코드는 수정하지 않고 결함만 드러냈습니다.** 수정 여부는 사용자 결정 사항입니다.

## 3. T2 / T3 / T4 — 이미 견고한 부분

### T2 교차 테넌트 격리 (통과, 20/20)
org A 토큰으로 org B의 operation을 조회하면 HTTP **404**가 반환되고, store의 `opSnapshot(orgID, id)`도 orgID 불일치 시 "없음"을 반환합니다. RLS 경계(모든 조회에 org_id 필터)가 코드 레벨에서 실제로 동작함을 확인했습니다. "남의 자원에 과금/유출" 벡터는 이 계층에서 차단됩니다.

### T3 미터링 원장 정합성 (통과, 20/20)
10개 org가 각각 20개 프로젝트를 **동시에** 생성(총 200 유효 요청, 멱등키 전부 고유)한 뒤, org별 `branch_ops` 미터링을 집계한 결과 **모든 org가 정확히 20건**, 전체 원장 합계 **정확히 200건**이었습니다. append-only 원장의 동시 기록이 손실·중복 없이 org별로 정확히 분리됩니다. 이는 `addMetering`이 뮤텍스로 보호된 단일 연산이고, 집계가 org_id로 필터링되기 때문입니다. (T1과의 차이: T3는 각 요청이 서로 다른 멱등키라서 TOCTOU 창이 열리지 않음)

### T4 원자적 온보딩 (통과)
장애주입용 테스트 로컬 mock(`failingKms`, `failingPayment`)을 `App`에 주입해 실패를 강제:
- **KMS 실패:** `Signup`이 KEK 프로비전 단계에서 즉시 에러 반환, org **0건** 생성.
- **billing 실패:** billing은 org 객체 생성 이후·`store.put` 이전에 호출되므로, 실패 시 에러 반환되고 org는 store에 **적재되지 않아 0건**. 고아 리소스 없음.

즉 현재 `Signup`의 순서(KEK → org객체 → billing → **store.put** → membership → token)는 "store에 넣기 전에 실패 지점들을 모두 통과"하는 구조라서, 이 두 실패 시나리오에서는 부분 커밋이 발생하지 않습니다. **단, 주의:** `store.put(org)` **이후** 단계(membership 추가, 토큰 발급)에서 실패하면 org만 남는 부분 커밋이 가능합니다. 이번 T4는 그 경로까지는 주입하지 않았으므로, 후속 검증 대상으로 남깁니다.

## 4. 검증의 교훈 — "PASS를 믿지 마라"

T1은 **첫 실행에서 우연히 통과(PASS)**했습니다. 만약 거기서 멈췄다면 "정합성 OK"라는 잘못된 결론을 내렸을 것입니다. 반복 실행(50회)으로만 4% 결함률이 드러났습니다.

> **원칙:** 동시성·정합성 테스트는 단발 PASS로 판정하지 않는다. 반드시 반복 실행(N회)하고, `-race`와 캐시 무효화를 병행하며, "왜 통과/실패했는지"를 코드로 설명할 수 있어야 한다.

이 원칙은 설계서의 모든 "0" 주장에 동일하게 적용되어야 합니다.

## 5. 재현 방법

```bash
cd sovereign_core
export PATH=$PATH:/usr/local/go/bin

# 전체 Stage 2 (T1은 비결정적으로 실패가 드러남)
go test -race -run TestStage2 -v ./...

# T1 이중과금 결함률 측정(캐시 무효화 50회)
for i in $(seq 1 50); do go test -run TestStage2_T1 -count=1 -v ./... 2>&1 | grep 'op='; done

# T2/T3/T4 견고성(반복해도 통과)
go test -run 'TestStage2_T[234]' -count=1 -v ./...
```

빌드/정적검사: `go build ./...` rc=0, `go vet ./...` rc=0 (테스트 추가 후에도 유지).
